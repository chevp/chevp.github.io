Synth Protocol: Native Szenen-Beschreibung

Das synth-protocol ist ein XML-basiertes Format fuer Projekt- und Szenen-Definitionen. Der Cluster Synth Editor versteht dieses Format nativ.

dashboard Der Cluster Synth Editor

Der Cluster Editor ist die native Arbeitsumgebung fuer synth-protocol Dateien. Er laedt .synth Dateien direkt und stellt ClusterNodes, Components und MCP-Tools visuell dar.

Direkte Verarbeitung

  • + Oeffnet .synth Dateien nativ
  • + Rendert ClusterNode-Hierarchie
  • + Zeigt Component-Daten
  • + Verbindet zu MCP-Servern

Kein Zwischenschritt

  • + Kein HTML-Export noetig
  • + Kein Protobuf-Parsing
  • + Direkte XML-Bearbeitung
  • + Version Control friendly

code Das .synth XML Format

Eine .synth Datei ist ein XML-Dokument mit definierten Namespaces. Es beschreibt Projekte, Szenen, Assets und MCP-Server-Konfigurationen.

Grundstruktur:

<?xml version="1.0" encoding="UTF-8"?>
<synthProject
    xmlns="https://chevp.github.io/synth-protocol/schema/synth/project/1.0"
    xmlns:core="https://chevp.github.io/synth-protocol/schema/synth/core/1.0"
    xmlns:mcp="https://chevp.github.io/synth-protocol/schema/synth/mcp/1.0">

    <header>
        <magic>SYNTH</magic>
        <schemaVersion>1.0.0</schemaVersion>
    </header>

    <project>
        <name>Mein Projekt</name>
        <settings>
            <defaultSceneId>scene_main</defaultSceneId>
        </settings>
    </project>

    <mcpServers>...</mcpServers>
    <clusterNodes>...</clusterNodes>
    <components>...</components>
    <assets>...</assets>

</synthProject>

account_tree ClusterNodes: Die Szenen-Hierarchie

ClusterNodes bilden den Szenen-Graph. Jeder Node hat einen Typ (Scene, Folder, Document, Asset) und kann Children sowie Components haben. 

<clusterNodes>
    <!-- Scene Node (Root) -->
    <node>
        <core:entityId>
            <uuid>scene_main</uuid>
        </core:entityId>
        <name>Main Scene</name>
        <nodeType>CLUSTER_NODE_TYPE_SCENE</nodeType>
        <childIds>
            <id>folder_models</id>
            <id>asset_hero</id>
        </childIds>
    </node>

    <!-- Asset Node (3D Modell) -->
    <node>
        <core:entityId>
            <uuid>asset_hero</uuid>
        </core:entityId>
        <name>hero_character.glb</name>
        <nodeType>CLUSTER_NODE_TYPE_ASSET</nodeType>
        <parentId>scene_main</parentId>
        <metadata>
            <entry key="format">glTF</entry>
            <entry key="vertex_count">125000</entry>
        </metadata>
    </node>
</clusterNodes>

widgets Components: Daten an Nodes

Components enthalten die eigentlichen Daten (Text, JSON, Bilder als Base64). Sie werden an ClusterNodes gehaengt - typisches ECS-Pattern. 

<components>
    <!-- Scene Settings als JSON -->
    <component>
        <core:entityId>
            <uuid>comp_scene_settings</uuid>
        </core:entityId>
        <clusterNodeId>scene_main</clusterNodeId>
        <componentType>scene_settings</componentType>
        <data encoding="json">{
            "gravity": {"x": 0, "y": -9.81, "z": 0},
            "ambient_color": {"r": 0.1, "g": 0.1, "b": 0.15}
        }</data>
    </component>

    <!-- Transform Component -->
    <component>
        <clusterNodeId>asset_hero</clusterNodeId>
        <componentType>transform</componentType>
        <data encoding="json">{
            "position": {"x": 0, "y": 0, "z": -5},
            "rotation": {"x": 0, "y": 45, "z": 0},
            "scale": {"x": 1, "y": 1, "z": 1}
        }</data>
    </component>

    <!-- Bild als Base64 -->
    <component>
        <clusterNodeId>asset_texture</clusterNodeId>
        <componentType>image_content</componentType>
        <data encoding="base64" mimeType="image/png">
            iVBORw0KGgoAAAANSUhEUgAA...
        </data>
    </component>
</components>

settings_input_component MCP-Server Integration

MCP-Server werden direkt in der .synth Datei konfiguriert. Der Cluster Editor startet sie automatisch und verbindet die Tools mit der UI (Ribbon). 

<mcpServers>
    <server>
        <serverName>synth-plugin-blender</serverName>
        <version>1.0.0</version>
        <required>true</required>
        <config>
            <parameters>
                <entry key="blender.executable">/usr/bin/blender</entry>
            </parameters>
            <autoStart>true</autoStart>
        </config>
    </server>
</mcpServers>

<!-- Ribbon: MCP Tools in der UI -->
<ribbon>
    <tabs>
        <tab>
            <label>Blender</label>
            <mcpServerId>mcp_blender</mcpServerId>
            <groups>
                <group>
                    <label>Export</label>
                    <tools>
                        <tool>
                            <mcpToolName>blender/export/gltf</mcpToolName>
                            <label>Export glTF</label>
                            <icon>3d_rotation</icon>
                        </tool>
                    </tools>
                </group>
            </groups>
        </tab>
    </tabs>
</ribbon>

web Web Output Pipeline

Fuer Web-Export rendert der Cluster Editor (via C++ Vulkan) die Szenen zu Assets und generiert Web Components. Die .synth Datei bleibt die Source of Truth.

Native Workflow

Cluster Editor arbeitet direkt mit .synth → Vulkan Pre-Rendering → Web Component Export

Output

Sprite-Sheets, Videos mit Alpha, leichtgewichtige Custom Elements 

<!-- Generiertes Web Component aus .synth Asset -->

    data-scene="scene_main"
    data-sprite="assets/cryo/hero_360.webp"
    data-frames="72"
></cryo-viewer>
